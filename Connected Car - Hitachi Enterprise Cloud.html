Prereqs

You will need the demo_key.pem file for SSH
You will need the root user password for the DC/OS cluster
You will need to elastic user password for the Kibana UI
Contact Don Kamer, Luke Miller, or Chris Stone for the above assets

You cannot access the application UI from the Hitachi Vantara corporate network, including VPN.  The required ports are blocked.  Since you can only access this wiki from the corporate network you should keep this page open and disconnect before accessing the UI.  Don't blame me, I'm just the messenger!  :) 
The DC/OS cluster must be rebuilt between demonstrations of the Connected Car Demo.

Prior to the Customer Demonstration

SSH to the DC/OS Bootstrap machine which we will use to launch application workload on the DC/OS cluster: 

ssh -i demo_key.pem centos@216.84.237.132

Make sure you are authenticated to the cluster from the DC/OS Bootstrap machine: 

dcos auth login

If you don’t have the DC/OS CLI installed on the node you are running CLI commands from go here.  This should rarely be the case from the bootstrap node.

If, and only if, the connectedcar folder on the bootstrap has been refreshed or recreated you will need to replace some tokens in the connected car application configuration. Find and replace our IP Addresses in the configuration JSON: 

cp connected-car-config.jsontemplate config.tmp &&
sed -ie "s@PUBLIC_IP_TOKEN@216.84.237.93@g;" config.tmp &&
sed -ie "s@CLUSTER_URL_TOKEN@https://216.84.237.75@g;" config.tmp

The above is rarely needed!

Install packages for the data services the connected car application will depend on. The Kibana service takes about 10 minutes to completely install so it's a good idea to have that done in advance to save dead time during the customer presentation: 

cd $HOME/connectedcar &&
dcos package install --options=kibana_config.json --yes beta-kibana --package-version=1.0.13-5.4.1-beta &&
cd $HOME

Wait 10 minutes for the service to complete. In the UI it should appear as healthy, or green. You can login to the DC/OS UI as the root user and click on the Services link on the left side to see the status of the Kibana install. Once Kibana is green your environment is ready for the customer presentation.

During the Customer Demonstration

Install the other data service dependencies: 

cd $HOME/connectedcar &&
dcos package install --yes marathon-lb --package-version=1.10.0 &&
dcos package install --yes cassandra --package-version=1.0.25-3.0.10 &&
dcos package install --yes kafka --package-version=1.1.19.1-0.10.1.0 &&
dcos package install --yes beta-elastic --package-version=1.0.13-5.4.1-beta --options=elastic_config.json &&
cd $HOME

You can login to the DC/OS UI as the root user and click on the Services link on the left side to see the status of the data services installations. Discussion Point: 

Now is a good time to discuss the value of using the turnkey data services to accelerate time to value for application developers and operations teams.

Create marathon group using the JSON at the command line: 

dcos marathon group add $HOME/connectedcar/config.tmp

Discussion Point: 

This command deploys a group of applications needed to stitch the existing data services into a data pipeline and visualize the data in a meaningful way.
Discuss the implications for Developer Agility to be able to group discreet services together and deploy them as an application.  This is enables micro-service application architectures to be much more easily managed for both the application and the operations team.

During the previous discussion, monitor for group install to complete.

Start the simulators for the "Cars", or "IOT devices" reporting data into the application we've just created  

 dcos marathon app add $HOME/connectedcar/cc_actor.json

Discussion point: 

You can login to the DC/OS UI as the root user and click on the Services link on the left side to see the status of the Actors install.  
This is a good opportunity to discuss scaling applications.  You can click on the ellipsis next to the Actors service and click Scale, run the count up to 10 and click OK.  Watch the deployment count scale up instantaneously.  Later in the UI we will show that we have data for 10 actors.

Take the customer through the application we've just deployed and loaded up with data: 

Browse to he Application UI
Click on Kafka and click START to see the raw data from the actors coming in to Kafka
Click on CQL and click RUN to see the data after it is transformed through Elastic and loaded into Cassandra in more structured format that can be queried.
Click on Dashboard and login with the elastic user account then click on the bottom button in the upper left corner of the map to center the map on the data.

Discussion Point: 

Highlight the green bars below the map, there should now be 10 of them since we scaled up the number of Actors being simulated on the DC/OS cluster.  The point is to show the actions being taken in the demo are really impacting the application and load we are running in the demo.
The map is now visualizing the data from the simulated cars driving around the Bay Area.  This was all possible with minimal work from the developers and operations teams to make the application run on the platform.  The operational, deployment, and development costs are radically lower due to our full managed, curated, and secure container orchestration platform.
